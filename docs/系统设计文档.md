# 校园影院购票与排片管理系统系统设计文档

## 一、系统概述

### 1.1 项目名称

校园影院购票与排片管理系统

### 1.2 项目背景

随着校园文化活动的丰富，校园影院逐渐成为学生课余生活的重要组成部分。传统的校园影院管理方式通常依赖人工登记、线下排片和现场购票，存在信息更新不及时、排片管理不方便、订单统计困难等问题。

为了提高校园影院的管理效率，方便学生查看电影信息、选择场次和在线购票，本系统设计并实现了一个基于前后端分离架构的校园影院购票与排片管理系统。

### 1.3 项目目标

本系统的主要目标是实现校园影院日常业务的信息化管理，主要包括：

1. 管理员能够维护电影信息。
2. 管理员能够维护影厅信息。
3. 管理员能够创建和管理电影排片。
4. 用户能够浏览电影列表和电影详情。
5. 用户能够查看电影对应场次。
6. 用户能够选择座位并创建订单。
7. 用户能够模拟支付订单。
8. 管理员能够在后台查看、支付、取消和删除订单。

## 二、需求分析

### 2.1 用户角色分析

本系统主要包括两类用户角色：

| 用户角色 | 说明 |
|---|---|
| 普通用户 | 浏览电影、查看电影详情、选择场次、选座购票、模拟支付 |
| 管理员 | 管理电影、管理影厅、管理排片、管理订单 |

### 2.2 功能需求分析

#### 2.2.1 普通用户功能

普通用户主要使用前台页面，功能包括：

1. 查看电影列表  
   用户进入系统首页后，可以查看当前影院中的电影信息。

2. 查看电影详情  
   用户可以点击电影卡片中的“查看详情”按钮，进入电影详情页，查看电影的详细信息。

3. 查看电影场次  
   在电影详情页中，用户可以查看该电影的排片场次，包括影厅、放映时间、票价和状态。

4. 选座购票  
   用户可以选择某一场次，进入选座页面，选择座位并提交订单。

5. 模拟支付  
   用户创建订单后，可以选择模拟支付，系统将订单状态更新为“已支付”。

#### 2.2.2 管理员功能

管理员主要使用后台管理页面，功能包括：

1. 电影管理  
   管理员可以新增、编辑、删除和查看电影信息。

2. 影厅管理  
   管理员可以新增、编辑、删除和查看影厅信息。

3. 排片管理  
   管理员可以为电影安排放映场次，并维护排片信息。

4. 排片冲突校验  
   系统会检查同一影厅在同一时间段内是否存在排片冲突，避免同一影厅同时安排多个电影场次。

5. 订单管理  
   管理员可以查看订单列表，并对订单进行支付、取消和删除操作。

### 2.3 非功能需求分析

| 需求类型 | 说明 |
|---|---|
| 易用性 | 页面布局清晰，按钮功能明确，便于用户操作 |
| 可维护性 | 前后端分离，代码结构清晰，便于后期维护 |
| 可扩展性 | 后期可以继续扩展用户登录、真实支付、座位锁定等功能 |
| 安全性 | 后端接口对关键业务数据进行校验 |
| 稳定性 | 系统能够完成基本的电影、影厅、排片、订单管理流程 |

## 三、系统总体设计

### 3.1 系统架构

本系统采用前后端分离架构。

前端使用 Vue 3 负责页面展示和用户交互；后端使用 ASP.NET Core Web API 提供接口服务；数据库使用 SQL Server 存储系统数据；前后端通过 Axios 调用 RESTful API 进行数据交互。

系统架构如下：

```text
用户浏览器
   │
   │ 访问页面
   ↓
Vue 3 前端页面
   │
   │ Axios 请求
   ↓
ASP.NET Core Web API 后端
   │
   │ Entity Framework Core
   ↓
SQL Server 数据库
```

### 3.2 技术架构

| 层次 | 技术 | 说明 |
|---|---|---|
| 前端展示层 | Vue 3、Element Plus | 实现系统页面和用户交互 |
| 前端路由层 | Vue Router | 管理前台和后台页面路由 |
| 前端请求层 | Axios | 调用后端 Web API |
| 后端接口层 | ASP.NET Core Web API 9.0 | 提供 RESTful 接口 |
| 数据访问层 | Entity Framework Core | 操作 SQL Server 数据库 |
| 数据存储层 | SQL Server | 存储电影、影厅、排片和订单数据 |
| 接口测试 | Swagger | 测试和查看后端接口 |

### 4.2 前台用户模块设计

#### 4.2.1 电影列表模块

该模块用于展示系统中的电影信息。用户进入首页后，系统向后端发送请求，获取电影列表并展示在页面中。

主要功能：

- 展示电影名称
- 展示电影类型
- 展示导演
- 展示片长
- 展示评分
- 提供“查看详情”入口

#### 4.2.2 电影详情模块

该模块用于展示电影详细信息和该电影的排片场次。

主要功能：

- 展示电影基本信息
- 展示电影简介
- 展示排片场次
- 提供“选座购票”入口

#### 4.2.3 选座购票模块

该模块用于用户选择座位并创建订单。

主要功能：

- 展示电影名称
- 展示影厅名称
- 展示放映时间
- 展示票价
- 展示座位按钮
- 支持选择多个座位
- 自动计算订单金额
- 提交订单

#### 4.2.4 模拟支付模块

订单创建成功后，用户可以选择模拟支付。系统调用后端支付接口，将订单状态从“待支付”修改为“已支付”。

### 4.3 后台管理模块设计

#### 4.3.1 电影管理模块

电影管理模块主要用于维护电影基础数据。

管理员可以执行：

- 新增电影
- 编辑电影
- 删除电影
- 查看电影列表

#### 4.3.2 影厅管理模块

影厅管理模块主要用于维护校园影院中的影厅信息。

管理员可以执行：

- 新增影厅
- 编辑影厅
- 删除影厅
- 查看影厅列表

#### 4.3.3 排片管理模块

排片管理模块用于给电影安排放映场次。

管理员新增排片时，需要选择电影、影厅、开始时间、结束时间和票价。

系统需要校验：

1. 结束时间必须晚于开始时间。
2. 票价必须大于 0。
3. 电影必须存在。
4. 影厅必须存在。
5. 同一影厅不能在同一时间段内重复排片。

#### 4.3.4 订单管理模块

订单管理模块用于维护用户购票后生成的订单。

管理员可以执行：

- 查看订单列表
- 支付待支付订单
- 取消待支付订单
- 删除订单

## 五、业务流程设计

### 5.1 用户购票业务流程

```text
用户进入首页
      ↓
查看电影列表
      ↓
点击查看详情
      ↓
查看电影场次
      ↓
点击选座购票
      ↓
选择座位
      ↓
提交订单
      ↓
系统生成待支付订单
      ↓
用户模拟支付
      ↓
订单状态变为已支付
```

### 5.2 管理员排片业务流程

```text
管理员进入后台
      ↓
维护电影信息
      ↓
维护影厅信息
      ↓
进入排片管理
      ↓
选择电影和影厅
      ↓

      ↓
系统检查排片时间冲突
      ↓
无冲突则保存排片
      ↓
用户可在前台查看该场次
```

### 5.3 订单管理业务流程

```text
用户提交订单
      ↓
系统生成订单号
      ↓
订单状态为待支付
      ↓
管理员进入订单管理页面
      ↓
查看订单信息
      ↓
可执行支付、取消或删除操作
```

## 六、数据库设计

### 6.1 数据库概述

本系统使用 SQL Server 数据库存储业务数据。数据库主要包含四张核心表：

| 表名 | 说明 |
|---|---|
| Movie | 电影信息表 |
| Hall | 影厅信息表 |
| Schedule | 排片信息表 |
| Order | 订单信息表 |

### 6.2 实体关系说明

各实体之间关系如下：

```text
Movie 1 ─── n Schedule
Hall  1 ─── n Schedule
Schedule 1 ─── n Order
```

说明：

1. 一部电影可以有多个排片。
2. 一个影厅可以安排多个排片。
3. 一个排片可以产生多个订单。
4. 一个订单对应一个具体排片。

## 九、关键业务规则设计

### 9.1 排片时间规则

排片新增或修改时，需要满足：

1. 结束时间必须晚于开始时间。
2. 同一影厅在同一时间段内不能有多个排片。
3. 票价必须大于 0。
4. 排片必须关联真实存在的电影和影厅。

### 9.2 订单规则

订单创建时，需要满足：

1. 必须选择排片。
2. 必须至少选择一个座位。
3. 排片状态必须为正常。
4. 系统根据票价和座位数量自动计算总金额。
5. 新订单默认状态为待支付。
6. 支付后订单状态变为已支付。
7. 已支付订单不能再次支付或取消。
